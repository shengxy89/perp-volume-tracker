## Project Overview

**Name**: `perp-volume-tracker`
**Purpose**: Track the daily ratio of Hyperliquid vs Binance platform-level total perpetual futures 24h trading volume, store historical data, generate a chart + summary table, and email an HTML report every day.
**Hosting**: GitHub Actions (scheduled cron job), data stored as CSV committed back to the repo.

---

## Functional Requirements

### 1. Data Collection

- **Hyperliquid**: Call `POST https://api.hyperliquid.xyz/info` with body `{"type": "metaAndAssetCtxs"}`. Sum the `dayNtlVlm` field across ALL assets to get total platform 24h perpetual volume (USD notional).
- **Binance**: Call `GET https://fapi.binance.com/fapi/v1/ticker/24hr`. Sum the `quoteVolume` field across ALL symbols to get total platform 24h perpetual volume (USD notional).
- Both APIs are public and require NO API keys.
- **Timestamp**: Record the UTC+8 (HKT) date of the run as the data point date.

### 2. Ratio Calculation

- `ratio = hyperliquid_total_24h_volume / binance_total_24h_volume`
- Round ratio to 6 decimal places.
- Round volumes to 2 decimal places (USD).

### 3. Data Storage

- Store data in a **CSV file** at `data/volume_history.csv` in the repo.
- CSV columns: `date,hl_volume,bn_volume,ratio`
  - `date`: format `YYYY-MM-DD`
  - `hl_volume`: Hyperliquid total 24h volume in USD
  - `bn_volume`: Binance total 24h volume in USD
  - `ratio`: hl_volume / bn_volume
- On each run:
  1. Read existing CSV (if exists).
  2. Check if today's date already has an entry; if yes, **overwrite** it (idempotent).
  3. If not, append a new row.
  4. Write CSV back, sorted by date ascending.
- The GitHub Action will commit and push this updated CSV after each run.

### 4. Chart Generation

- Generate a **line chart** of the historical ratio using `matplotlib`.
- **Adaptive window**:
  - If total data points <= 30, show ALL history.
  - If > 30, show only the most recent 30 days.
- Chart specs:
  - Title: "Hyperliquid / Binance Perp Volume Ratio"
  - X-axis: date (formatted nicely, rotated 45 degrees)
  - Y-axis: ratio
  - Grid enabled, light style
  - Mark the latest data point with a red dot and annotate its value.
  - If today's ratio is a **historical all-time high** (compared to ALL history, not just display window), add a "ATH" annotation in red.
  - If today's ratio is a **historical all-time low** (compared to ALL history), add a "ATL" annotation in blue.
  - Also plot a light dashed horizontal line for the 7-day moving average.
- Save chart as `data/ratio_chart.png`.

### 5. Email Report

- Send an **HTML email** with the chart embedded inline (base64-encoded image) and a summary data table.
- **Email subject**: `[Perp Tracker] {date} | Ratio: {ratio} {alert_flag}`
  - `alert_flag`: append "ATH!" or "ATL!" if applicable, otherwise empty.
- **HTML body structure**:
  - Header: "Perp Volume Ratio Daily Report" with date
  - Summary Table with these rows:
    - Hyperliquid 24h Volume: formatted as $X,XXX,XXX,XXX
    - Binance 24h Volume: formatted as $XX,XXX,XXX,XXX
    - Ratio (HL/BN): 0.XXXXXX
    - 7-Day Avg Ratio: 0.XXXXXX (or "N/A (X days of data)" if insufficient)
    - 30-Day Avg Ratio: 0.XXXXXX (or "N/A (X days of data)" if insufficient)
    - vs Yesterday: +X.XX% with up/down arrow (or "N/A" if no previous data)
    - Status: "ATH" (red background) / "ATL" (blue background) / "Normal"
  - Inline Chart Image (base64 embedded PNG)
  - Footer: "Auto-generated by perp-volume-tracker"
- Format volumes with **comma separators** and `$` prefix.
- **Highlight rules in HTML**:
  - If ratio is all-time high across full history: Status row background light red, bold "ALL-TIME HIGH".
  - If ratio is all-time low across full history: Status row background light blue, bold "ALL-TIME LOW".

### 6. Alert Email on Failure

- If either API call fails (timeout, HTTP error, parsing error):
  - Still attempt the other API (don't fail both).
  - If at least one fails, send a **separate alert email** to the same recipient:
    - Subject: `[Perp Tracker] Warning: Data Collection Error — {date}`
    - Body: plain text describing which API failed, the error message, and HTTP status code if available.
  - Do NOT write incomplete data to CSV. Skip the data point for that day.

### 7. Configuration

- All configurable values via **environment variables** (for GitHub Actions secrets):
  - `SMTP_SERVER` (default: `smtp.gmail.com`)
  - `SMTP_PORT` (default: `587`)
  - `SMTP_USER` — sender email address
  - `SMTP_PASSWORD` — app password
  - `NOTIFY_EMAIL` — recipient email address (comma-separated for multiple)
  - `TZ` — timezone (default: `Asia/Hong_Kong`)
- Provide a `.env.example` file listing all variables with placeholder values.

## GitHub Actions Workflow (`daily_track.yml`)

```yaml
name: Daily Perp Volume Track

on:
  schedule:
    - cron: '0 1 * * *'    # 01:00 UTC = 09:00 HKT
  workflow_dispatch:         # allow manual trigger

permissions:
  contents: write            # needed to push CSV updates

jobs:
  track:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repo
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Run tracker
        env:
          SMTP_SERVER: ${{ secrets.SMTP_SERVER }}
          SMTP_PORT: ${{ secrets.SMTP_PORT }}
          SMTP_USER: ${{ secrets.SMTP_USER }}
          SMTP_PASSWORD: ${{ secrets.SMTP_PASSWORD }}
          NOTIFY_EMAIL: ${{ secrets.NOTIFY_EMAIL }}
          TZ: Asia/Hong_Kong
        run: python -m src.main

      - name: Commit updated data
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add data/
          git diff --cached --quiet || git commit -m "Update volume data $(date +%Y-%m-%d)"
          git push
```

## Technical Constraints

- **Python 3.11+**
- **Dependencies** (keep minimal): `requests`, `pandas`, `matplotlib`
- **No API keys needed** for either Hyperliquid or Binance public endpoints.
- All HTTP requests must have a **timeout of 30 seconds** and **retry up to 3 times** with exponential backoff (1s, 2s, 4s).
- Use `logging` module with level INFO; log to stdout (GitHub Actions will capture it).
- The entire `main.py` run should complete within **5 minutes** max.
- Code must be **idempotent**: running twice on the same day overwrites (not duplicates) the data point.
